AppLovin RTB supports the OpenRTB 2.5 API on ad requests from apps that MAX mediates.
This document describes the Open<Abbr>RTB</Abbr> integration with AppLovin for <Glossary>SDK</Glossary> <Glossary>bidders [in-app bidders]</Glossary> and <Glossary>DSPs</Glossary> and details any feature differences between the Open<Abbr>RTB</Abbr> 2.5 spec and the AppLovin <Abbr>RTB</Abbr> spec.

<Callout type="note">
<Glossary>GDPR</Glossary> non-consent and <Abbr>CCPA</Abbr> do-not-sell requests are <em>not</em> supported and will not be sent in the bid stream.
</Callout>

### Protocol Details

Bid Requests
: AppLovin <Abbr>RTB</Abbr> buyers receive bid requests in <Abbr>HTTP</Abbr> `POST` format.
  AppLovin recommends that you use <Abbr>HTTP</Abbr> so as to avoid <Glossary>latency</Glossary>.

<Abbr>HTTP</Abbr> Keep-Alive
: AppLovin <Abbr>RTB</Abbr> buyers should enable persistent connection handling.
  Set max keep-alive to at least five minutes.

Compressed/gzip <Abbr>oRTB</Abbr> 
: AppLovin <Abbr>RTB</Abbr> requires compression for efficiency.
  Adhere to [Open<Abbr>RTB</Abbr> <Abbr>API</Abbr> Specification version 2.5, section 2.4: Data Encoding](https://www.iab.com/wp-content/uploads/2016/03/OpenRTB-API-Specification-Version-2-5-FINAL.pdf#%5B%7B%22num%22%3A53%2C%22gen%22%3A0%7D%2C%7B%22name%22%3A%22XYZ%22%7D%2C69%2C229%2C0%5D).

### Bid Regions

The AppLovin <Abbr>RTB</Abbr> servers cover four geographical regions:

- ASIA-PACIFIC: Hong Kong, Tokyo
- EUROPE: Amsterdam
- US-EAST: New York, Washington
- US-WEST: San Francisco, San Jose

### Endpoints

<Abbr>DSP</Abbr> buyers and <Abbr>SDK</Abbr> bidders can specify up to one endpoint per region.
All requests from the respective region are routed to that endpoint.

### Supported Ad Formats

AppLovin <Abbr>RTB</Abbr> supports mobile <Glossary>in-app ads</Glossary> only; it does not support desktop or web ad serving.
The ad formats AppLovin <Abbr>RTB</Abbr> supports are <Glossary>banners</Glossary>, <Glossary>medium rectangles</Glossary> (<Abbr>MREC</Abbr>s), <Glossary>native ads</Glossary>, <Glossary>interstitials</Glossary>, and <Glossary>rewarded ads</Glossary>.

Acceptable <Abbr>MIME</Abbr> types include:

| ad format | <Abbr>MIME</Abbr> type |
| --------- | ---------------------- |
| <Abbr>MRAID</Abbr>, <Abbr>HTML</Abbr>, <Abbr>MREC</Abbr> | `text/html` |
| Native ads | `image/png`, `image/jpeg`, `image/gif` |
| <Abbr>VAST</Abbr> | `video/mp4` |

Acceptable <Abbr>MRAID</Abbr> and <Abbr>VAST</Abbr> versions include those in the following list.
However, there are limitations.
For example, AppLovin <Abbr>RTB</Abbr> does not support calendar <Abbr>API</Abbr> events from <Abbr>MRAID</Abbr>.

 - <Abbr>MRAID</Abbr> 1.0
 - <Abbr>MRAID</Abbr> 2.0
 - <Abbr>MRAID</Abbr> 3.0
 - <Abbr>VAST</Abbr> 2.0
 - <Abbr>VAST</Abbr> 3.0
 - <Abbr>VAST</Abbr> 4.0
 - <Abbr>VAST</Abbr> 4.1
 - <Abbr>VAST</Abbr> 4.2

There is additional information further down on this page concerning <Abbr>MRAID</Abbr>, native ads, and <Abbr>VAST</Abbr> support.

### Exchange Currency

AppLovin <Abbr>RTB</Abbr> Exchange supports only <Abbr>USD</Abbr>.
It ignores the currency attribute `cur`.

### Auction Type

MAX auction runs on a first-price basis.
AppLovin expects bidders to pay the <Glossary>CPM</Glossary> amount they bid in full.

### Blocked Creative Types

AppLovin <Abbr>RTB</Abbr> blocks the following <Glossary>creative</Glossary> types:

 - Adobe Flash
 - Audio Ad (Auto-Play)
 - Audio Ad (User Initiated)
 - Expandable (User Initiated — Rollover)
 - Pop (e.g. Over, Under, or Upon Exit)
 - Provocative or Suggestive Imagery
 - Windows Dialog or Alert Style

If the publisher does not support <Glossary>playable ads</Glossary>, then AppLovin <Abbr>RTB</Abbr> also blocks the following creative types:

 - Expandable (Automatic) [only for non-banner/MREC]
 - Expandable (User Initiated — Click) [only for non-banner/MREC]
 - User Interactive (e.g. Embedded Games)

### Blocked Categories

AppLovin <Abbr>RTB</Abbr> blocks the following <Abbr>IAB</Abbr> categories:

 - `IAB8-5` (Cocktails/Beer)
 - `IAB8-18` (Wine)
 - `IAB11-4` (Politics)
 - `IAB25` (Non-Standard Content)
 - `IAB26` (Illegal Content)

### Blocked Ad Types

AppLovin <Abbr>RTB</Abbr> blocks the following ad types:

 - flash
 - iframe
 - <Abbr>XHTML</Abbr> Text Ad

## Bid Request
### Request Objects

AppLovin <Abbr>RTB</Abbr> bid requests always carry a single <Glossary>impression</Glossary> (this differs from the Open<Abbr>RTB</Abbr> schema).
Each request has exactly one impression object (`imp`).

### Seats

The AppLovin <Abbr>RTB</Abbr> Exchange does not support multiple seats per bidder.
It ignores the seat whitelist `wseat` array.

### Impression Object

An AppLovin <Abbr>RTB</Abbr> bid request always carries a single impression (this differs from the Open<Abbr>RTB</Abbr> schema).
Each request has exactly one impression object (`imp`).

If the request includes a banner object in the impression object, the impression is eligible for <Abbr>HTML</Abbr> and <Abbr>MRAID</Abbr> ads.
If the request includes a video object, the impression is eligible for <Abbr>VAST</Abbr> ads.
The impression object may either include only a banner or video sub-object, or it may include both such objects.

## Bid Response
### Size

The maximum size of the bid response is 4<Abbr>KB</Abbr>.

### No Bids

To communicate a no-bid response, pass an empty response with a status code of 204.

### Win/Loss/Billing Notifications

AppLovin <Abbr>RTB</Abbr> calls the win, loss, and billing notice <Abbr>URL</Abbr>s that the requester supplies in the request’s `nurl`, `lurl`, and `burl` fields respectively.
AppLovin <Abbr>RTB</Abbr> calls the win <Abbr>URL</Abbr> when the auction completes and AppLovin sends the ad to the device.
It calls the loss <Abbr>URL</Abbr> either during the auction, if the associated ad fails to load properly, or after a different winning ad loads.
It calls the billing <Abbr>URL</Abbr> when the device renders the ad—that is, when the video starts playing, or, for graphic ads, when the device displays the ad.

The requester may embed macros into the specification of these notification <Abbr>URL</Abbr>s.
AppLovin <Abbr>RTB</Abbr> substitutes appropriate values for those macros in the <Abbr>URL</Abbr> before AppLovin <Abbr>RTB</Abbr> requests the <Abbr>URL</Abbr>.
The following list shows the macros that the requester may embed in these <Abbr>URL</Abbr>s:

These macros are case-sensitive; specify them in ALL CAPS.
Enclose these macros in `${…}` (e.g. `${AUCTION_ID}`).

| Macro                        | Description |
| ---------------------------- | ----------- |
| `AUCTION_BID_ID`             | <Abbr>ID</Abbr> of the bid, from the `BidResponse.bidid` attribute. |
| `AUCTION_ID`                 | <Abbr>ID</Abbr> of the bid request, from the `BidRequest.id` attribute. |
| `AUCTION_LOSS`               | Loss reason code, as found in [Table 5.25](https://www.iab.com/wp-content/uploads/2016/03/OpenRTB-API-Specification-Version-2-5-FINAL.pdf#%5B%7B%22num%22%3A171%2C%22gen%22%3A0%7D%2C%7B%22name%22%3A%22XYZ%22%7D%2C69%2C152%2C0%5D) of the Open<Abbr>RTB</Abbr> 2.5 spec. Only sent on `lurl`. Note that loss code “103 Lost to a Bid for a <Glossary>PMP</Glossary> Deal” is used when the impression is won by a Direct Sold campaign at a higher priority. |
| `AUCTION_MINIMUM_BID_TO_WIN` | Second highest bid price. If there is no winner, floor price is sent. Is only sent on `nurl`. Value passed is `0` when there is only one bid: the winning bid. |
| `AUCTION_PRICE`              | The auction clearing price. Uses the same currency and units as the bid. May be up to six decimals. |

#### Example Win Bid <Abbr>URL</Abbr> with Macros

```plaintext
https://example.com/endpoint/auction=${AUCTION_ID}&price=${AUCTION_PRICE}
```

### Ad Markup Standard

<Abbr>DSP</Abbr> bidders must include ad markup in the bid response. AppLovin <Abbr>RTB</Abbr> does not support returning the ad markup in response to the win notice.

Your ad markup must comply with AppLovin’s <Abbr>RTB</Abbr> ad markup standard.
AppLovin’s <Abbr>RTB</Abbr> servers check the ad markup’s compliance as soon as they receive the bid, and then they perform the following sanity checks on the ad markup:

- Ad markup is properly escaped.
- Ad markup is <Abbr>URL</Abbr>-encoded, except for <Abbr>VAST</Abbr> ads.
- <Abbr>HTML</Abbr> of the ad markup is a snippet.
- When you bid with a rich media ad, put the content of the node in a `CDATA` construct.
  `CDATA` in <Abbr>XML</Abbr> requires no encoding whatsoever.
  However if you pass an <Abbr>XML</Abbr> document in a <Abbr>JSON</Abbr>-context, you must apply <Abbr>JSON</Abbr> syntax and escaping rules, as follows:
  - All double quotes are escaped as `\"`.
  - Apostrophes are not escaped.
  - The <Abbr>XML</Abbr> doc is stripped of all tab and newline characters.
    ([The Example <Abbr>VAST</Abbr> Ad Markup](#vast-support) below is formatted for easy readability.
    Do not send extra whitespace in your actual ad markup.)

To verify that an ad markup snippet is valid <Abbr>JSON</Abbr> content (along with its `CDATA`), you can pass that snippet into [the on-line <Abbr>JSON</Abbr>Lint validator](https://jsonlint.com/).

<Abbr>SDK</Abbr> bidders are not required to pass ad markup because all creatives should be rendered via your <Abbr>SDK</Abbr>.
<Abbr>SDK</Abbr> bidders may use the `admarkup` field to pass custom data, which can be configured to be returned to your <Abbr>SDK</Abbr> at load time via the adapters.

### <Abbr>VAST</Abbr> Support

<Abbr>VAST</Abbr> videos auto-play on Android and <Abbr>iOS</Abbr>.
AppLovin supports only Linear videos.
Videos should be at least 5 seconds long and at most 30 seconds long.
Video file sizes should be below 4<Abbr>MB</Abbr> (for optimal playback, AppLovin recommends video files to be smaller than 2<Abbr>MB</Abbr>).
AppLovin aspect-fits the video in the middle of the screen, which depends on the device size.

The `adm` (or ad markup field) must contain the entire <Abbr>VAST</Abbr> 2.0 <Abbr>XML</Abbr> payload, <Abbr>VAST</Abbr> 3.0 <Abbr>XML</Abbr> payload, or <Abbr>VAST</Abbr> 4.0 <Abbr>XML</Abbr> payload.
AppLovin does not support <Abbr>URL</Abbr>-encoded <Abbr>VAST</Abbr> tags. Embed <Abbr>VAST</Abbr> in `bid.adm`.

#### Impression Tracking

AppLovin fires tags on the server side that you send through the `<impression>` node. You can send more than one such node.

#### Click Tracking

AppLovin fires the tag on the client side that you send through the `<ClickTracking>` node.
If you use a <Glossary>deep link</Glossary>, you must include it in the `<ClickThrough>` node.

If you use a deep-link in an <Abbr>iOS</Abbr> ad, the `ext.skadn` object must be `null`.
Otherwise the App Store will be opened within the ad and the `<ClickThrough>` node will be dismissed.

#### Companion Ads

AppLovin <Abbr>RTB</Abbr> expects a full-screen companion ad in the response for <Abbr>VAST</Abbr> ads.
The request specifies the phone screen dimensions.
The AppLovin <Abbr>SDK</Abbr> aspect-fits the ad in the middle of the screen.
Since these are video end cards, use something in the ~1.33 aspect ratio range (for example, 1024×768 for landscape) and high quality images for best results with companion ads.


AppLovin supports only `<HTMLResource>` and `<StaticResource>`.
Support for `<HTMLResource>` as a companion ad is off by default.

### <Abbr>MRAID</Abbr> Support

<Abbr>MRAID</Abbr> ads must include a loading indicator to signal that the ad is loading.
There is no limit on ad duration for regular <Abbr>MRAID</Abbr> ads, but rewarded <Abbr>MRAID</Abbr> ads have a maximum of 30<Abbr>s (seconds)</Abbr>.
AppLovin supports `mraid.open` only for click trackers.

Rewarded <Abbr>MRAID</Abbr> placements (playable ads) are off by default.
If you are interested in these, please contact your AppLovin point of contact.

#### Close Button

Remove any close button from your <Abbr>MRAID</Abbr> ad.
AppLovin supplies the close button on all <Abbr>MRAID</Abbr> ads.

### <Abbr>HTML</Abbr> Support

AppLovin <Abbr>RTB</Abbr> centers all <Abbr>HTML</Abbr> ads.
The resulting ad position depends on the ad size and the screen size.
AppLovin supplies the close button.

### <Abbr>CTV</Abbr> Support

AppLovin <Abbr>RTB</Abbr> supports Connected TV (<Abbr>CTV</Abbr>) inventory.
The following will help with a successful integration: 

- <Abbr>CTV</Abbr> requests support video only, so there is no `banner` object under [the `imp` object](#imp-object).
- <Abbr>VAST</Abbr> is the only supported format for Connected <Abbr>TV</Abbr> inventory.
  AppLovin does not support <Abbr>MRAID</Abbr> or <Abbr>VPAID</Abbr>.
- Only the mp4 video <Abbr>MIME</Abbr> type is supported.
- [`seatbid.bid.ext.duration`](#bid-object) is mandatory.

For <Abbr>VAST</Abbr> InLine:

- The bitrate should fall between [`minbitrate` and `maxbitrate`](#video-object) sent in [the bid request](#bid-request).
- The width and height should be equal to [`imp.video.w` and `imp.video.h`](#video-object) in [the bid request](#bid-request).

### Ad Caching

The AppLovin <Abbr>SDK</Abbr> caches certain ads to improve ad display responsiveness.
Cached ads are shown only in the same session.
The <Abbr>SDK</Abbr> caches ads for up to four hours, but normally the period between cache and show is only a few minutes.

Video
: AppLovin caches only inline video ads.
  For optimal responsiveness, the maximum video file size allowed is 4MB.
  Try to keep video file sizes below 2MB.

<Abbr>MRAID</Abbr>
: AppLovin does not cache <Abbr>MRAID</Abbr> ads.
  While AppLovin does not have a size limit, keep your creative assets in your <Abbr>MRAID</Abbr> markup as lean as possible to ensure your ads are responsive when the publisher attempts to render them.
  All <Abbr>MRAID</Abbr> ads must include a loading indicator to signal when the ad is loading.
  The AppLovin close button appears after three seconds, regardless of loading status.
